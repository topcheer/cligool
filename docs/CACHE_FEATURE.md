# CLI 消息缓存功能

## 功能概述

当没有 Web 客户端连接时，中继服务器会自动缓存 CLI 客户端发送的所有消息。当 Web 客户端连接时，会先发送缓存的历史消息，然后实时广播后续消息。

## 使用场景

这个功能解决了以下问题：
- 用户在本地终端执行命令后，分享连接给其他人
- 其他人打开 Web 界面时，能够看到之前执行的命令和输出
- 避免用户误以为连接有问题

## 工作原理

### 消息流程

```
CLI 客户端 ──消息──> 中继服务器
                      │
                      ├─ 无 Web 客户端 ──> 缓存消息
                      │
                      └─ 有 Web 客户端 ──> 实时广播

Web 客户端连接 ──> 发送缓存消息 ──> 实时广播后续消息
```

### 缓存策略

1. **缓存触发条件**：当 `session.Clients` 为空（没有 Web 客户端）时
2. **缓存大小限制**：最多缓存 1000 条消息
3. **缓存淘汰策略**：FIFO（先进先出），缓存满时删除最旧的消息
4. **缓存内容**：消息类型（type）和数据（data）

### 内存管理

- 预分配 1000 条消息的容量
- 跟踪缓存总大小（字节）
- 自动删除最旧的消息，避免内存无限增长

## 代码实现

### Session 结构

```go
type Session struct {
    // ... 原有字段
    MessageCache   []CachedMessage // CLI消息缓存
    CacheSizeLimit int            // 缓存大小限制（1000条）
    TotalCacheSize int            // 缓存总大小（字节）
}
```

### 关键函数

#### `addToCache()`
- 将消息添加到缓存
- 检查缓存大小限制
- 自动淘汰最旧的消息
- 记录日志

#### `sendCachedMessages()`
- 发送所有缓存的消息给新连接的 Web 客户端
- 使用只读锁，避免阻塞其他操作
- 记录发送进度

#### `broadcastToWebClients()`
- 检查是否有 Web 客户端连接
- 如果没有，调用 `addToCache()` 缓存消息
- 如果有，实时广播消息

## 性能考虑

### 内存占用

假设平均每条消息 100 字节：
- 最大缓存：1000 条 × 100 字节 = 100 KB
- 每个会话的缓存开销很小

### 性能影响

- **缓存添加**：O(1) 时间复杂度（slice append）
- **缓存淘汰**：O(1) 时间复杂度（删除第一个元素）
- **缓存发送**：O(n) 时间复杂度（n = 缓存消息数）
  - 只在 Web 客户端连接时执行一次
  - 不影响实时广播性能

## 未来改进

1. **持久化缓存**：将缓存保存到磁盘，支持跨重启恢复
2. **压缩缓存**：使用压缩算法减少内存占用
3. **可配置限制**：允许用户自定义缓存大小限制
4. **选择性缓存**：只缓存特定类型的消息（如 output）
5. **缓存过期**：自动删除过期的缓存消息
